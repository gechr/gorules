# gorules

Shared [ruleguard](https://github.com/quasilyte/go-ruleguard) lint rules for Go, consumed across repos through [go-critic](https://github.com/go-critic/go-critic)'s `ruleguard` checker under [golangci-lint](https://golangci-lint.run).

## Bundles

The rules split across two packages by kind, because the two kinds must run in different golangci-lint passes (see below):

- **`github.com/gechr/gorules`** - **diagnostics** (`Report` rules). Run in the lint gate.
- **`github.com/gechr/gorules/format`** - **formatters** (`Suggest` rules). Run only as a `--fix` pass.

| Rule             | Bundle   | Flags                                                                                                                                                                                                                                                  |
| ---------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `stringEquality` | root     | `assert`/`require` `Contains`/`NotContains` on a **string** (slice/map membership is left alone), and any `ErrorContains` - use `assert.Equal` on the full string, or `assert.ErrorIs` for errors.                                                     |
| `clogchain`      | `format` | A `clog` builder chain (`.With().Str(...).Int(...)....Logger()`) with two or more field calls - wraps it to one call per line, regardless of width. Generated by `format/gen/clog.go`; run `make gen` after bumping the max chain length in that file. |

**Why the split.** A `Suggest` rule matches on AST shape, not on whether the code is already laid out the way its suggestion would rewrite it (newlines are not in the AST), so it re-reports on *already-formatted* code. In a plain `golangci-lint run` that surfaces as a permanent finding on correct code. Formatters therefore must run only as a `--fix` formatting pass, never in the lint gate - so they live in their own bundle, imported by a dedicated config. A single `dsl.ImportRules` call imports a whole package's rules, so keeping formatters out of the lint gate means keeping them in a separate package.

## Usage

Each bundle needs its own loader and its own golangci-lint config. Put both loaders in a non-dot `gorules/` directory guarded by a `//go:build ruleguard` tag: the tag keeps them out of normal compilation (`go build`/`vet`), while the non-dot path stays visible to `go mod tidy` so it does **not** prune the `gorules` require. (A dot-directory like `.ruleguard/` is invisible to the whole toolchain, so `go mod tidy` would drop the require and silently disable the rules.)

### Diagnostics (lint gate)

```go
//go:build ruleguard

// gorules/rules.go
package gorules

import (
	bundle "github.com/gechr/gorules"
	"github.com/quasilyte/go-ruleguard/dsl"
)

func init() { dsl.ImportRules("gechr", bundle.Bundle) }
```

Wire it into `.golangci.yml` (the `rules:` path is required - golangci-lint does not auto-discover it):

```yaml
linters:
  settings:
    gocritic:
      enabled-checks:
        - ruleguard
      settings:
        ruleguard:
          rules: gorules/rules.go
```

### Formatters (`--fix` pass)

```go
//go:build ruleguard

// gorules/format.go
package gorules

import (
	format "github.com/gechr/gorules/format"
	"github.com/quasilyte/go-ruleguard/dsl"
)

func init() { dsl.ImportRules("gechr", format.Bundle) }
```

Wire it into a **dedicated** `.golangci.ruleguard.yml` that enables only `ruleguard`, and invoke it with `--fix` from `make fmt` (never from `make lint`):

```yaml
version: "2"
linters:
  default: none
  enable:
    - gocritic
  settings:
    gocritic:
      disable-all: true
      enabled-checks:
        - ruleguard
      settings:
        ruleguard:
          failOn: all
          rules: gorules/format.go
```

```makefile
fmt:
	@golangci-lint run --fix -c .golangci.ruleguard.yml
```

Both bundles resolve from the module cache, so a single `require github.com/gechr/gorules` in the consumer's `go.mod` covers the subpackage too. The build-tagged loaders keep `go mod tidy` from pruning that require.
